# Service Scope Template

| Scope Area | Included Items | Owner | Notes |
| --- | --- | --- | --- |
| In-scope teams | Teams assessed, locations, time zones |  |  |
| In-scope backlogs | Product, sprint, defect, technical debt backlogs |  |  |
| In-scope repositories | Repositories reviewed for process evidence, not code audit unless agreed |  |  |
| Sprint history reviewed | Number of sprints and date range |  |  |
| Stakeholder interviews | Roles and approximate count |  |  |
| Delivery metrics | Velocity, throughput, cycle time, lead time, escaped defects, carryover |  |  |
| Releases examined | Recent releases, hotfixes, incidents, rollback history |  |  |
| AI-development workflows | Approved prompts, tooling, review gates, traceability artifacts where permitted |  |  |
| Explicit exclusions | Teams, repos, data, tools, legal/security certifications outside scope |  |  |

## Scope-Change Process

1. Document requested change, reason, and decision deadline.
2. Assess impact to scope, schedule, cost, evidence needs, and stakeholders.
3. Obtain written approval before beginning changed work.
4. Update the proposal, SOW, roadmap, and evidence plan if approved.
